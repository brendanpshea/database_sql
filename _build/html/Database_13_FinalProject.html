Learning Outcomes#

By the end of this project, you will be able to:

  1. Translate a real-world scenario into business rules, entities, attributes, relationships, and an ERD.

  2. Convert the conceptual model into a normalized relational design with appropriate keys and data types.

  3. Implement the schema in SQLite with DDL and populate it with realistic sample data.

  4. Write basic and advanced queries that demonstrate filtering, joins, aggregation, and analysis.

  5. Create views and indexes that improve usability or performance, and explain the trade-offs.

  6. Demonstrate transaction management and safe data modification practices.

  7. Explain design choices, testing approach, and optimization decisions in clear written documentation.

Aim for a project that takes roughly 6 to 9 hours to complete: challenging enough to be meaningful, but still manageable.

Part 21: Project Documentation Packet#

A real database project is judged on more than its tables and queries. The team that has to run the system after you turns the documentation packet first. For this final project, your packet must include four short pieces.

1. Standard Operating Procedure (SOP)#

One-page document describing the routine maintenance for your database. At minimum:

  • How to take a backup, and how often it should run.

  • How to confirm the backup completed successfully.

  • How to rebuild or re-analyze indexes if performance drifts.

  • One health-check query the next operator can run each morning.

Write the SOP so a teammate could follow it without asking you any questions.

2. Organizational compliance note#

One paragraph identifying the regulations that would apply if your project were a real system. Examples:

  • A patient-records project would face HIPAA.

  • A student-records project would face FERPA.

  • A payment-processing project would face PCI-DSS.

  • A project handling EU users would face GDPR.

Name the regulation, say what data triggers it, and list one specific control you would put in place to comply (encryption at rest, audit logging, access reviews, etc.).

3. Third-party compliance note#

One paragraph naming any vendor or partner agreements the project would need. Examples:

  • A cloud provider Business Associate Agreement (BAA) for HIPAA data.

  • A data processing agreement (DPA) with an analytics vendor under GDPR.

  • A PCI-DSS attestation from a payment processor.

If your project would not involve a third party, say so explicitly and explain why.

4. Maintenance documentation entry#

A short note describing how schema changes will be tracked over time. At minimum:

  • Where migrations or change scripts will live (version control, a shared folder, a wiki).

  • Who reviews and approves schema changes.

  • How rollbacks would work if a migration goes wrong.

Together, the four pieces form a documentation packet a reviewer can use to evaluate the project as a system, not just a set of working queries.

Glossary#

Use this reference while you complete the final project.

  • Business rule — A short statement describing how the system should behave or what it must enforce.

  • Entity — A real-world object or concept that you store data about.

  • Attribute — A property of an entity, such as a name, date, or price.

  • Relationship — A connection between two entities.

  • ERD — An Entity Relationship Diagram that shows entities, their attributes, and the relationships among them.

  • Primary key — A column that uniquely identifies each row in a table.

  • Foreign key — A column that points to a related row in another table.

  • Normalization — Organizing tables to reduce redundancy and improve data integrity.

  • Schema — The overall structure of a database, including its tables, columns, and constraints.

  • DDL — Data Definition Language, the SQL commands that create or change database structure, such as CREATE TABLE and ALTER TABLE.

  • INSERT — A SQL statement that adds new rows to a table.

  • SELECT — A SQL statement that retrieves data from one or more tables.

  • JOIN — A SQL operation that combines related rows from two or more tables.

  • Aggregate function — A function such as COUNT(), SUM(), or AVG() that summarizes many rows into a single value.

  • Index — A data structure that speeds up data retrieval for common queries.

  • EXPLAIN QUERY PLAN — A SQLite command that shows how the database plans to run a query.

  • View — A saved SQL query that behaves like a virtual table.

  • Transaction — A sequence of operations treated as one unit of work.

  • COMMIT — A SQL command that makes the changes in a transaction permanent.

  • ROLLBACK — A SQL command that undoes the changes in a transaction if something goes wrong.

  • Standard operating procedure (SOP) — A short document describing how to perform a routine operational task, written so a new team member could follow it.

  • Organizational compliance — Internal and external rules the organization is required to follow, including industry regulations and contractual obligations.

  • Third-party compliance — Compliance obligations that arise from vendor or partner relationships, often documented in agreements such as a BAA or DPA.

  • Maintenance documentation — A record of how a database’s schema and operational changes are tracked, reviewed, and rolled back.
